# Writeup Cash

Bug bounty is just the beginning. Real money comes from monetizing your knowledge: writeups, tools, courses, consulting. One good writeup can generate $500-$5000 in passive income.

## When to Use

- Have accepted bug bounty reports to share
- Want to build passive income from security research
- Building a personal brand in security
- Creating content for paid platforms
- Developing and selling security tools

## The Process

1. **Scope the task** — define objectives, boundaries, and success criteria
2. **Gather information** — collect all necessary data and context before proceeding
3. **Execute the core workflow** — follow the domain-specific steps methodically
4. **Validate results** — verify outputs against expected outcomes or baselines
5. **Document findings** — record results, anomalies, and recommendations

#### Writeup Structure
```markdown
# [Vulnerability Type] in [Target] - $[Bounty Amount]

## TL;DR
One paragraph summary of the finding, impact, and payout.

## Target
- Application: [Name]
- Platform: [HackerOne/Bugcrowd]
- Scope: [In-scope assets]

## Discovery
How you found the vulnerability:
- Recon process
- What caught your attention
- Initial hypothesis

## Vulnerability Details
Technical explanation:
- What's wrong
- Why it's wrong
- How it can be exploited

## Proof of Concept
Step-by-step reproduction:
1. [Step with screenshot]
2. [Step with code/commands]
3. [Step with result]

## Impact
Business impact:
- Who is affected
- What data is at risk
- Financial/reputational impact

## Remediation
How to fix:
- Specific code changes
- Configuration changes
- Architecture recommendations

## Lessons Learned
What you learned:
- New technique discovered
- Common pattern to look for
- Tool/script that helped

## Timeline
- Discovery: [Date]
- Report: [Date]
- Triage: [Date]
- Fix: [Date]
- Bounty: [Date]
```

#### Sanitization Checklist
```
- [ ] Remove all real user data
- [ ] Redact internal URLs/IPs
- [ ] Remove API keys/tokens
- [ ] Blur sensitive screenshots
- [ ] Get permission from program before publishing
- [ ] Wait for fix before public disclosure
- [ ] Follow program's disclosure policy
```

## When NOT to Use

- Task is outside your authorization scope
- You need to implement controls (use implementing-* skills)
- Task is about analysis, not action (use analyzing-* skills)
- You don't have access to target systems
- Task requires compliance expertise (consult professionals)
- Task is about defense, not offense (use defensive skills)


## Red Flags

- Publishing writeups before fix is deployed
- Sharing real user data in writeups
- Overstating severity for attention
- Copying other hunters' work
- Not crediting collaborators
- Selling tools with known vulnerabilities

## Verification

- All writeups are sanitized and approved by program
- Tools are tested and documented
- Content is original and accurate
- Pricing is competitive and sustainable
- Brand is consistent across platforms
- Income streams are diversified
